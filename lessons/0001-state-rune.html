Lesson 1 · Svelte 5 fundamentals

$state — reactive variables that feel like Python

A 10-minute lesson. By the end you will know exactly what let count = $state(0) does, why it re-renders the page, and the one rule that catches every newcomer.

Why this lesson exists

Your project atopic_helper is built on Svelte 5. Every interactive page in it — the questionnaire, the day view, the meal log — is held together by a tiny idea called a rune. The most important rune is $state. Get it, and $derived, $effect, and props are easy. Miss it, and you will spend hours debugging "why didn't this update?".

Look at line 30 of src/routes/+page.svelte in your project. You wrote (or inherited): let step = $state(1). This lesson is the explanation that line deserves.

The mental model

In plain JavaScript, let count = 0 is a number. If you change it, nothing in the world notices — the variable just holds a new value.

In Svelte 5, let count = $state(0) looks identical, but the compiler quietly rewrites it. Reading count now subscribes the surrounding component to that value, and writing to it notifies every subscriber to re-run. The component re-renders. You did not write any callbacks.

Python feels like

An attribute on a traitlets-style object that auto-fires observe() on change — except you don't have to register the observer. The framework does it.

Rust feels like

A Cell<T> wrapped in a signal: interior mutability + a publish/subscribe channel. The compiler inserts the get/set calls so you don't see them.

Source: svelte.dev/docs/svelte/$state

The smallest working example

<!-- Counter.svelte -->
<script>
  let count = $state(0);
</script>

<button onclick={() => count++}>
  Clicked {count} times
</button>

Three things are happening:

  1. $state(0) creates a reactive container holding 0.
  2. The {count} in the markup reads it — that read registers a dependency.
  3. Clicking runs count++, which writes a new value — Svelte schedules a re-render of every place that read it.
JS basics — for the curious

let declares a re-assignable variable; const would forbid count++. The arrow function () => count++ is JS shorthand for function() { return count++ }. You'll see arrows everywhere; treat them like Python lambdas. onclick= is JSX-style attribute binding new in Svelte 5 (older Svelte used on:click).

The one rule that catches everyone

When $state wraps an array or object, Svelte makes it deeply reactive using a JavaScript Proxy. That means mutating in place works:

let motherAllergies = $state([]);

// All of these trigger re-render:
motherAllergies.push('milk');
motherAllergies[0] = 'egg';
motherAllergies.length = 0;

So why does your +page.svelte do this instead?

motherAllergies = motherAllergies.filter((a) => a !== single);
motherAllergies = [...motherAllergies, single];

Because immutable updates — "build a new array, assign it" — are easier to reason about, and they play better with $derived (next lesson). Both styles work; the spread/filter style is the codebase's convention. Pick a style, stick with it.

The trap

If you destructure a reactive value out of $state, the copy is no longer reactive. let { name } = user gives you a frozen snapshot. Always read through the original: user.name. Same applies to passing a single field as a prop — pass the parent object or wire it explicitly.

Reading your own code (no editing)

Open ../atopic_helper/src/routes/+page.svelte in your editor for reading only. Find these lines near the top of the <script> block:

let step = $state(1);
let babyBirthDate = $state('');
let severity = $state<EczemaSeverity>('moderate');
let motherAllergies = $state<string[]>([]);

Each line declares a piece of component-local reactive state. The <EczemaSeverity> bit is a TypeScript type argument — "this state holds an EczemaSeverity value." It's a label for the type-checker; at runtime it disappears. The behaviour is identical to the JS-only versions.

The same patterns are mirrored verbatim in playground/src/routes/lesson/state/+page.svelte — that is where you'll experiment.

Try it (5 minutes) — in the playground, not your real project

The playground is a tiny SvelteKit app at ../playground/ in this workspace. Your atopic_helper code is read-only reference; we never poke it.

  1. In a terminal: cd ~/Developer/svelte_learning/playground && bun run dev
  2. Open http://localhost:5180/lesson/state in a browser.
  3. Click both buttons. The reactive counter updates; the plain counter's display freezes — even though the click handler does run (the log proves it).
  4. Open playground/src/routes/lesson/state/+page.svelte. Find let reactiveCount = $state(0). Remove the $state(...) wrapper so it reads let reactiveCount = 0. Save. The "reactive" side now behaves like the plain side — that is the rune disappearing.
  5. Put the wrapper back. Now try the opposite: wrap plainCount with $state. Both counters now update.
  6. Bonus: log.push(...) in the click handler works because log is a deeply-reactive $state array. Try replacing log = [] in reset() with log.length = 0 — same effect, both work.

You can now read every $state line in your project and predict what it does. That is roughly half of Svelte 5's reactivity model. The other half — $derived (computed values) and $effect (side-effects) — is built on top of this same idea, and is the next lesson.
