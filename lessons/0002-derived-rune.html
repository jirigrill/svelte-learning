$derived — computed values that recompute themselves

A 12-minute lesson. By the end you will read const progress = $derived(((step - 1) / (TOTAL_STEPS - 1)) * 100) in your own codebase and know exactly what it does, when it runs, and what it costs.

The problem $derived solves

Imagine you have a questionnaire with 6 steps. The user is on step 3. You want a progress bar showing 40%. The 40% isn't state — it's a calculation: ((step - 1) / (TOTAL_STEPS - 1)) * 100. So how do you keep it in sync with step?

Three approaches, increasingly bad to good:

❌ Approach 1: store it in $state and update by hand

let step = $state(1);
let progress = $state(0); // derived from step, but stored separately

function nextStep() {
  step++;
  progress = ((step - 1) / (TOTAL_STEPS - 1)) * 100; // remember to update!
}

This is the bug factory. Anywhere step changes, you must also remember to recompute progress. Forget once and your progress bar lies. This is cache invalidation, the famously hardest problem in computing.

❌ Approach 2: a regular function call in the markup

let step = $state(1);
function calcProgress() { return ((step - 1) / (TOTAL_STEPS - 1)) * 100; }

<!-- markup -->
<ProgressBar value={calcProgress()} />

This works — Svelte sees step being read inside the function call and re-renders. But it has no caching: if calcProgress() appears in three places, it runs three times per render. For trivial maths that's fine; for sorting an array of 200 meals, it's a problem.

✅ Approach 3: $derived

let step = $state(1);
const progress = $derived(((step - 1) / (TOTAL_STEPS - 1)) * 100);

<!-- markup -->
<ProgressBar value={progress} />

One line. Always in sync with step. Memoised: only recomputes when step actually changes. Reads like English. This is the line in your +page.svelte.

Source: svelte.dev/docs/svelte/$derived

The mental model

$derived(expression) creates a computed signal. Svelte does three things behind the scenes:

  1. Tracks dependencies. The first time the expression runs, Svelte notes every $state it reads. $derived(((step - 1) / (TOTAL_STEPS - 1)) * 100) depends on step.
  2. Caches the result. Reading progress from anywhere returns the cached value — no recomputation.
  3. Invalidates lazily. When step changes, the cache is marked stale. Next time anyone reads progress, the expression re-runs and the new value is cached.

Python feels like

@cached_property on a class — the value is computed lazily and cached, but invalidated when its inputs change. Except in Svelte the invalidation is automatic, not manual.

Rust feels like

Like a memoised closure that captures references to other signals. The compiler builds the dependency graph for you; you write the formula.

Three rules to internalise

RuleWhy
Use const, never let. $derived values are read-only by design. You can't reassign them — that's the point. Trying to is a compile error. Same with $state: let is required there because it is mutable. The keyword tells you (and the compiler) the intent.
Keep the expression pure. No side effects. The expression may run zero, one, or many times depending on whether anyone reads it. Triggering an API call or DOM mutation here is unpredictable. For side effects you want $effect, the next lesson.
For multi-line logic, use $derived.by(() => { ... }). The single-expression form ($derived(x + y)) is convenient for one-liners. For complex computation with intermediate variables, $derived.by(() => { const a = …; const b = …; return a + b; }) gives you a function body. Same semantics; just syntax.

Source: svelte.dev/docs/svelte/$derived#$derived.by

Reading your own code

Open ../atopic_helper/src/routes/+page.svelte for reference (no editing). Find these:

const progress = $derived(((step - 1) / (TOTAL_STEPS - 1)) * 100);

const permanentSlugs = $derived(
  [...new Set([
    ...motherAllergies.map(s => s.split(':')[0]),
    ...babyAllergies.map(s => s.split(':')[0])
  ])]
);

Two derived values, two patterns:

  • progress — depends on step. Re-runs whenever the user advances the questionnaire.
  • permanentSlugs — depends on motherAllergies AND babyAllergies (both $state arrays). Re-runs when either changes. Notice the dependency tracking is automatic — you didn't have to declare "depends on these two" anywhere; Svelte saw the reads and wired it up.

Now look at ../atopic_helper/src/lib/components/AllergenChip.svelte line 25:

const display = $derived(resolveDisplay(slug));

This pattern is the workhorse: a derived value that calls a pure function with a reactive input. Whenever slug changes, resolveDisplay re-runs and display updates. The function itself doesn't need to know about reactivity.

Multi-input dependencies — automatic

You never write depends_on=[step] or anything like that. Svelte sees which signals are read while the expression evaluates and wires them up automatically. Add a new dependency to the formula? Tracked. Remove one? Untracked. This is one of the things runes get gloriously right.

Bonus: $derived cures the destructuring trap

Remember Lesson 1's trap?

let user = $state({ name: 'Jiri', clicks: 0 });
const { name } = user; // frozen snapshot — UI doesn't update on user.name changes

$derived gives us a live read instead:

let user = $state({ name: 'Jiri', clicks: 0 });
const name = $derived(user.name); // always current — recomputes when user.name changes

The expression user.name goes through the Proxy on every recompute — so it's always live. The "fence" you sensed in Lesson 1 (single source of truth) is now enforced by the framework, not by your discipline.

That said: if you only use name once in markup, just write {user.name} directly. $derived shines when (a) the expression is non-trivial, (b) it's used in multiple places, or (c) you want a clear local name for what the expression means.

Quick self-check

  1. What is the difference in intent between let count = $state(0) and const total = $derived(count * 2)?
  2. If count is read by 5 different $derived expressions, and count changes once, how many times does each derived expression recompute? (Hint: also think about whether anyone reads them.)
  3. Why is it a bad idea to write const x = $derived((() => { console.log('hi'); return count; })())?
  4. In atopic_helper, look at const phase = $derived(ctx.status === 'ready' ? getPhaseForDate(ctx.schedule, selectedDate) : null). Which two reactive sources does it depend on? What value does phase hold while ctx.status is, say, 'loading'?
The win

You can now read every $derived line in your project and predict (a) what its dependencies are, (b) when it recomputes, and (c) whether it's pulling its weight. Combined with Lesson 1, you've covered the inputs ($state) and the pure outputs ($derived) of the reactive graph. The remaining piece — side effects ($effect) — is Lesson 3.
